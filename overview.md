## 二、关键决策摘要

### 决策 1：MVP 阶段用飞书多维表格，不上数据库
- 原因：300 家企业规模飞书完全够用，0 代码、可分享、字段可视化。
- 切换时机：超过 1000 家或并发 > 5 人时切 PostgreSQL + PostGIS。

### 决策 2：评分模型 8 维度固定权重，不做动态学习
- 原因：MVP 阶段样本不足，动态学习容易过拟合。
- 演进：每月人工校验 30 家 → 季度调权重 → 半年后尝试机器学习。

### 决策 3：AI 输出全部结构化 + 可编辑
- 原因：销售不愿被 AI "绑架"，所有 AI 输出必须可改。
- 体现：跟进总结 / 话术 / KP 路线都支持"AI 生成 → 人工微调 → 落库"。

### 决策 4：KP 信息以角色为主，不强求姓名
- 原因：合规 + 数据可获得性。
- 体现：contacts.name 字段允许 NULL，contact_role 必填。

### 决策 5：话术库沉淀到独立表，迭代靠 effectiveness_score
- 原因：销售实战中话术效果差异巨大，必须用数据驱动优化。
- 体现：每次使用自动 +1 usage_count，每季度根据跟进结果更新 effectiveness_score。

## 五、风险与注意事项

### R1：采集速度可能不达预期
- 应对：第 1 周至少投入 1 名助理，专注录入。
- 兜底：100 家也能跑通 MVP，300 家是 stretch 目标。

### R2：AI 评分与销售判断可能不一致
- 应对：抽 30 家人工评分对比，校准提示词。
- 兜底：保留"人工覆盖"开关。

### R3：销售可能不愿主动用
- 应对：每天 23:00 推送"今日 Top 10"到企业微信。
- 兜底：先用 2 周跑通 3 家客户，再要求强制使用。

### R4：数据合规边界
- 严守：仅采集工商/官网/招聘/公开新闻信息。
- 严禁：个人手机号、私人微信、身份证号。

## 2.3 关键技术决策

| 决策 | 选择 | 原因 |
| - | - | - |
| 架构 | 静态站点 + 7 个独立 HTML | 不需后端、可独立分享、零运维 |
| 地图 | 自绘 SVG（基于 8 商区经纬度） | 避免腾讯地图 SDK 申请流程；MVP 够用 |
| 评分 | 8 维度加权，前端实时计算 | 与 PRD §8.1 完全一致；可视化展示给销售 |
| AI | 模板化生成 + 占位符 | 不依赖外部 API；可立即使用；后续替换为真实 LLM |
| 主题 | 浅色（premium glass + 渐变） | 与 IDE 浅色主题一致；后续可加 dark 切换 |
| 持久化 | localStorage | 单人 MVP 够用；不污染用户数据 |
| 部署 | CloudStudio 沙箱 | 公网 URL 可立即分享；后续可绑独立域名 |
